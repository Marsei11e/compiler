# semantics.md — Семантика языка

---

## 1. Стратегия вычисления

Язык использует **call-by-value**: при вызове функции все аргументы вычисляются до входа в функцию, и в функцию передаются копии значений. Изменение параметра внутри функции не влияет на переменную вызывающего кода.

Исключение — значения типа `range[T]`: они передаются по **перемещению** (move), а не по копированию (§7.5).

```
fn double(x: int32) -> int32 {
    x = x * 2;   -- изменяет локальную копию
    return x;
}

fn main() -> int32 {
    var a: int32 = 5;
    var b: int32 = double(a);
    -- a по-прежнему равно 5, b == 10
    return 0;
}
```

---

## 2. Семантика присваивания

Присваивание выполняется по значению (**value semantics**): правая часть копируется в левую. Для структур это означает побитовое копирование всех полей. Для массивов — копирование всех элементов.

После присваивания две переменные независимы — изменение одной не влияет на другую.

```
var p1: Point = Point{ x: 1.0, y: 2.0 };
var p2: Point = p1;   -- полная копия
p2.x = 99.0;          -- p1.x остаётся 1.0
```

Исключение: присваивание значения типа `range[T]` **перемещает** его (см. §7.5).

---

## 3. Инициализация переменных

Каждая переменная **обязана** быть инициализирована в момент объявления. Использование неинициализированной переменной является ошибкой компиляции.

```
var x: int32;          -- ошибка компиляции: нет инициализатора
var y: int32 = 0;      -- корректно
var z = 42;            -- корректно: тип выведен, значение инициализировано
```

Константа (`const`) также обязана быть инициализирована при объявлении и не может быть присвоена повторно.

```
const MAX: int32 = 100;
MAX = 200;             -- ошибка компиляции: присваивание константе

const name = "hello";  -- корректно: тип string выведен
```

Вывод типа для `var`/`const` без аннотации — см. `types.md`, §5.

---

## 4. Время жизни переменных

Переменная существует в пределах **блока**, в котором она объявлена. При выходе из блока переменная уничтожается (после выполнения всех `defer`-инструкций этого блока — §7.8).

```
fn main() -> int32 {
    var x: int32 = 1;
    {
        var y: int32 = 2;   -- y создаётся здесь
        x = x + y;
    }                        -- y уничтожается здесь
    -- использование y здесь — ошибка компиляции
    return x;
}
```

## 5. Область видимости и разрешение имён

Язык использует **лексическую область видимости** (lexical scoping).

### 5.1 Правила разрешения

- Идентификатор разрешается в ближайшей охватывающей области видимости.
- Использование идентификатора до его объявления не допускается — ошибка компиляции.
- Одно и то же имя не может быть объявлено дважды в одной области видимости — **исключение**: функции могут быть перегружены (§13).

### 5.2 Пространства имён идентификаторов

В языке существует несколько **разделённых** пространств имён:

1. **Обычные имена** — переменные, обычные функции, пространства имён, синонимы типов. В одной области видимости имя может принадлежать ровно одному из них (кроме перегруженных функций — §13).
2. **Имена типов-структур** — совпадают с обычными именами (структура `Point` и переменная `Point` конфликтуют).
3. **Пространство имён методов типа** — для каждого типа-структуры `T` имена методов и ассоциированных функций образуют отдельное пространство. Метод `distance` у `Point` не конфликтует с функцией `distance` на уровне модуля.

### 5.3 Перекрытие имён (shadowing)

Внутреннее объявление с тем же именем скрывает внешнее в пределах своего блока. Это применимо к переменным и не применимо к функциям (функции нельзя объявить во вложенных блоках).

```
var x: int32 = 1;
{
    var x: int32 = 2;   -- скрывает внешний x
    println(x);          -- выведет 2
}
println(x);             -- выведет 1
```

### 5.4 Пространства имён (namespaces)

Объявления внутри `namespace` доступны снаружи через оператор `::`. Внутри пространства имён обращение к собственным объявлениям идёт без префикса.

```
namespace Math {
    const PI: float64 = 3.14159;

    fn area(r: float64) -> float64 @pure {
        return PI * r * r;   -- PI без префикса
    }
}

fn main() -> int32 {
    var a: float64 = Math::area(5.0);   -- с префиксом
    return 0;
}
```

### 5.5 Разрешение методов и ассоциированных функций

Для выражения `expr.method(args)`:
1. Определяется тип `expr` — это должен быть тип-структура `T`.
2. В пространстве методов `T` ищется имя `method`.
3. С учётом перегрузок выбирается подходящая перегрузка по типам `(T, args)` (§13).
4. Если метод не найден — ошибка `no method named 'method' for type T`.

Для выражения `T::name(args)`:
1. Имя `T` должно разрешаться в тип-структуру.
2. В пространстве методов `T` ищется имя `name` (ассоциированная функция — без `self`).
3. Разрешение перегрузки — по типам `args`.

## 6. Семантика выражений

### 6.1 Арифметические операции

Операнды должны быть одного типа. Неявных приведений нет — если типы отличаются, требуется явный `cast`.

Деление целых чисел (`/`) — целочисленное (усечение к нулю). Остаток от деления (`%`) определён только для целых типов.

Деление целых чисел на ноль — **ошибка времени выполнения**:

```
runtime error: division by zero at line <N>
```

Деление вещественных чисел (`/` для `float32`/`float64`) **не** является ошибкой времени выполнения. Поведение полностью определяется стандартом IEEE 754:

- `x / 0.0` (где `x > 0`) — `+inf`;
- `x / 0.0` (где `x < 0`) — `-inf`;
- `0.0 / 0.0` — `NaN`.

Это означает, что float-деление **не несёт эффект `@panics`** (см. §12.2) и допустимо в теле `@pure`-функций. Для целочисленного `%` — поведение аналогично `/` (для целых — runtime error при делителе `0`; для float — `%` не определён).

### 6.2 Логические операции

`&&` и `||` используют **короткое замыкание** (short-circuit evaluation):
- `false && expr` — `expr` не вычисляется, результат `false`
- `true || expr` — `expr` не вычисляется, результат `true`

### 6.3 Сравнения

Операторы сравнения (`==`, `!=`, `<`, `>`, `<=`, `>=`) применимы к числовым типам и `bool`. Для строк допускаются только `==` и `!=` (лексикографическое сравнение байт).

Сравнение значений разных типов без явного приведения — ошибка компиляции.

### 6.4 Конкатенация строк

Оператор `+` для типа `string` выполняет конкатенацию:

```
var s: string = "Hello, " + "world!";
```

### 6.5 Индексирование массива

Индекс должен быть целочисленного типа. Выход за границы массива является **ошибкой времени выполнения**:

```
runtime error: index out of bounds at line <N>
```

Допустимые индексы: от `0` до `N - 1`, где `N` — размер массива, зафиксированный в типе.

### 6.6 Доступ к полям структуры

Обращение к несуществующему полю — ошибка компиляции.

### 6.7 Явное приведение типов

```
cast<T>(expr)
```

Выражение `expr` приводится к типу `T`. Допустимые приведения описаны в `types.md`, §7. Приведение к несовместимому типу — ошибка компиляции.

### 6.8 Диапазоны (`..`)

Выражение `a..b` (оба целочисленные, одного типа `T`) даёт значение типа `range[T]`. Оба операнда вычисляются слева направо **в момент создания диапазона**. Последующее изменение переменных, из которых был построен диапазон, не влияет на его границы — значения уже скопированы внутрь диапазона.

Диапазон — полуинтервал: `0..10` содержит `0, 1, 2, …, 9`.

### 6.9 Оператор `|>` (pipe)

`a |> f` семантически эквивалентно `f(a)`.  
`a |> f(b, c)` семантически эквивалентно `f(a, b, c)`.

Порядок вычисления:
1. Вычисляется `a`.
2. Вычисляются `b, c, ...` слева направо (если есть).
3. Вызывается `f` с этим набором аргументов.

Это соответствует естественному порядку вычисления вызова `f(a, b, c)`.

### 6.10 Условное выражение `if`

`if cond { A } else { B }`:
1. Вычисляется `cond` — обязан иметь тип `bool`.
2. Если `true` — вычисляется `A`, значение выражения = значение `A`.
3. Иначе — вычисляется `B`, значение выражения = значение `B`.

Ветки `A` и `B` обязаны иметь одинаковый тип.

### 6.11 Блочное выражение

`{ stmt1; stmt2; ...; stmtN; final_expr }`:
1. Создаётся новая область видимости.
2. Инструкции выполняются по порядку.
3. Если блок завершается выражением без `;` — его значение становится значением блока.
4. При выходе из блока выполняются все `defer` этого блока (§7.8).
5. При выходе — все локальные переменные уничтожаются.

Если блок в позиции выражения не содержит финального выражения, его тип — `hollow`, и значение отсутствует.

## 7. Семантика инструкций

### 7.1 Объявление переменной

1. Вычисляется инициализирующее выражение.
2. Если тип явно указан — проверяется совместимость с типом инициализатора (с учётом подстройки безсуффиксных литералов).
3. Если тип опущен — тип переменной устанавливается равным типу инициализатора (правила — `types.md`, §5).
4. Результат копируется в новую переменную (либо перемещается, если тип — `range[T]`).
5. Переменная доступна с этой точки до конца текущего блока.

### 7.2 Присваивание

Левая часть (`lvalue`) должна быть мутабельной переменной, элементом массива или полем структуры, хранящейся в мутабельной переменной. Присваивание константе — ошибка компиляции.

Для обычных типов — копирование. Для `range[T]` — перемещение (§7.5).

### 7.3 if / else

Условие обязано иметь тип `bool`. Ветка `else` необязательна — в этом случае `if` не является выражением (имеет тип `hollow`). `else if` записывается естественно.

```
if cond1 {
    ...
} else if cond2 {
    ...
} else {
    ...
}
```

### 7.4 while

Условие обязано иметь тип `bool`. Тело выполняется, пока условие истинно. Условие вычисляется перед каждой итерацией.

### 7.5 for

Синтаксис: `for IDENT in EXPR BLOCK`.

Семантика зависит от типа `EXPR`:

#### По диапазону (`range[T]`)

1. Выражение `EXPR` вычисляется один раз.
2. Если `EXPR` — переменная, она считается **потреблённой** после этой точки.
3. `IDENT` последовательно принимает значения `a, a+1, ..., b-1`.
4. Если `b <= a` — тело не выполняется ни разу.

```
var r = 0..5;
for i in r {     -- r потреблён
    println(i);
}
-- использование r здесь — ошибка компиляции
```

Диапазон-литерал, переданный напрямую в `for`, не требует переменной и не создаёт проблем:

```
for i in 0..5 { println(i); }
for i in 0..5 { println(i); }  -- корректно: два независимых временных диапазона
```

#### По массиву (`array[T, N]`)

1. Выражение `EXPR` вычисляется.
2. Создаётся **копия** массива (value semantics).
3. `IDENT` последовательно принимает значения копий `arr[0], arr[1], ..., arr[N-1]`.
4. Если `N == 0` — тело не выполняется. (В текущей версии языка `N > 0`, так что этот случай невозможен.)

В обоих случаях `IDENT` **иммутабельна** в теле цикла. Присваивание `IDENT = ...` внутри `for` — ошибка компиляции.

#### Move-семантика диапазонов

Тип `range[T]` не является копируемым. Формально:

- При инициализации `var r1 = r2;`, где `r2: range[T]`, значение **перемещается** из `r2` в `r1`. После этой точки `r2` считается потреблённым.
- При передаче аргумента `f(r)` — `r` потреблён.
- При использовании в `for IDENT in r` — `r` потреблён.
- При возврате `return r;` из функции — `r` потреблён (естественно, это последнее использование).

**Проверка на компиляции:** семантический анализатор ведёт для каждой локальной переменной типа `range[T]` флаг «потреблена». Любое использование потреблённой переменной — ошибка компиляции:

```
error: use of moved value 'r' at line N
```

Диапазон-литерал (rvalue) — это временное значение, и правило потребления на него не распространяется напрямую, кроме момента использования.

Условные потребления (разные ветки `if`/`for`) обрабатываются консервативно: если переменная потреблена хотя бы в одной ветке пути выполнения, после слияния ветвей она считается потреблённой. Пример:

```
var r = 0..10;
if cond {
    for i in r { println(i); }   -- потребляется здесь
}
-- здесь r считается потреблённой независимо от cond
for i in r { ... }               -- ошибка
```

### 7.6 break и continue

`break` — немедленно завершает ближайший охватывающий цикл (`while` или `for`).  
`continue` — переходит к следующей итерации ближайшего охватывающего цикла.

Перед фактическим выходом по `break` или переходом по `continue` выполняются **все активные `defer`-инструкции** всех покидаемых блоков в порядке LIFO (§7.8).

Использование `break` или `continue` вне цикла — ошибка компиляции.

### 7.7 return

В функции с типом возврата, отличным от `hollow`: `return expr` — тип `expr` должен совпадать с типом возврата функции (с учётом подстройки числовых литералов).

В функции с типом возврата `hollow`: `return` без выражения. Допускается также достижение конца тела функции без явного `return`.

Достижение конца тела функции с не-`hollow` типом возврата без `return` — ошибка компиляции.

Перед фактическим возвратом выполняются все активные `defer`-инструкции всех покидаемых блоков в порядке LIFO.

### 7.8 defer

`defer stmt` регистрирует инструкцию `stmt` к выполнению при выходе из содержащего её блока. Точная семантика:

**Регистрация:**
1. Инструкция `defer stmt` при выполнении **не выполняет** `stmt` немедленно.
2. Она добавляет `stmt` в конец списка отложенных действий текущего блока.
3. Если `defer` сам не был достигнут — его действие не регистрируется (не существует).

**Выполнение:**
Отложенные действия блока выполняются при любом выходе из блока:
- естественное завершение блока (достижение `}`);
- `return` внутри блока или из любого вложенного;
- `break` / `continue`, покидающие этот блок;
- паника (§10).

Отложенные действия выполняются в порядке, обратном регистрации (LIFO).

При выходе через несколько блоков одновременно (вложенные блоки покидаются одним `return`) отложенные действия каждого блока выполняются в порядке: сначала все действия самого внутреннего блока (LIFO), затем следующего по уровню (LIFO), и так до функции.

**Пример:**

```
fn main() -> int32 {
    defer println("1");
    {
        defer println("2");
        defer println("3");
        println("A");
    }
    println("B");
    return 0;
}
-- Вывод: A, 3, 2, B, 1 (по строкам)
```

**Ограничения:**
- `defer return ...` — ошибка компиляции (отложенный `return` нарушает семантику функции).
- `defer break` / `defer continue` — ошибка компиляции.
- `defer defer X` — ошибка компиляции.

Отложенные действия **не могут** отменить выход из блока; они выполняются «по пути».

Если отложенное действие само вызывает `panic`, поведение определено следующим образом: все ранее зарегистрированные `defer` всё равно выполняются (в порядке LIFO), последующая паника распространяется наверх.

### 7.9 Нулевая инструкция

Одиночная `;` не производит никакого эффекта.

## 8. Побочные эффекты

Функции могут производить побочные эффекты: вывод в stdout, чтение из stdin, изменение локальных переменных.

Порядок вычисления операндов и аргументов — слева направо, что обеспечивает предсказуемый порядок побочных эффектов.

Для функций, аннотированных эффектами, компилятор производит **статическую проверку** корректности эффектов (§12).

## 9. Встроенные функции

### `print(expr)`

Эффект: `@io`. Выводит строковое представление значения в стандартный поток вывода **без** добавления перевода строки. Принимает значение одного из **скалярных** встроенных типов: целочисленного (`int8`..`int64`, `uint8`..`uint64`), вещественного (`float32`, `float64`), `bool` или `string`. Составные типы (`array[...]`, структуры) и `range[...]` печатать нельзя — для них перегрузки `print`/`println` нет, вызов даёт ошибку `no matching overload`. Возвращает `hollow`.

### `println(expr)` и `println()`

Эффект: `@io`. Две перегрузки:

- `println(expr)` — выводит строковое представление значения и добавляет перевод строки (`\n`). Принимает значения тех же типов, что и `print` (скалярные числовые, `bool`, `string`). Возвращает `hollow`.
- `println()` — без аргументов, выводит только перевод строки. Возвращает `hollow`.

Семантически `println(x)` эквивалентно `print(x)` с последующим выводом `\n`, но реализуется как единый вызов рантайма для эффективности.

**Формат вывода.** Текстовое представление значения соответствует форматированию `printf` языка C:

- целые знаковые — десятичное (`%lld`), беззнаковые — десятичное (`%llu`); типы уже `int64`/`uint64` расширяются перед печатью;
- вещественные (`float32` расширяется до `double`) — формат `%.6f`: десятичная запись с **фиксированными 6 знаками после точки**, завершающие нули сохраняются (`2.0` → `2.000000`, `0.5` → `0.500000`, `78.53981633974483` → `78.539816`);
- `bool` — литералы `true` / `false`;
- `string` — байты строки без изменений (UTF-8).

> **Замечание.** Поскольку безсуффиксный числовой литерал подходит сразу под несколько перегрузок `print`/`println` (§13.2), печать «голого» литерала — `println(1)`, `println(3.14)` — даёт ошибку `ambiguous call`. Нужен суффикс (`println(1L)`), `cast` или печать значения через переменную/результат выражения известного типа.

### `input() -> string`

Эффект: `@io`. Читает одну строку из стандартного потока ввода (до символа новой строки, не включая его). Возвращает `string`.

### `exit(code: int32)`

Эффект: `@panics` (в смысле прерывания потока управления). Немедленно завершает программу. Значение `code` передаётся как код завершения процесса. Не возвращает управление. Перед завершением активные `defer` **не выполняются** — это отличие `exit` от естественного возврата.

### `panic(msg: string)`

Эффект: `@panics`. Аварийно завершает программу. Выводит в stderr:

```
runtime error: <msg> at line <N>
```

Перед завершением выполняются все активные `defer`-инструкции всех покидаемых блоков (как при `return`), в порядке LIFO. Не возвращает управление. Код завершения процесса — `1`.

### `len(s: string) -> int32`

Эффект: `@pure`. Возвращает количество байт в строке.

## 10. Поведение при ошибках времени выполнения

При возникновении ошибки времени выполнения программа завершается следующим образом:

1. В stderr выводится сообщение:
   ```
   runtime error: <сообщение> at line <N>
   ```
2. Активные `defer`-инструкции всех блоков текущей функции (и функций в стеке вызовов) выполняются в порядке LIFO — для корректной очистки ресурсов.
3. Программа завершается с кодом `1`.

Обязательно обрабатываются:

| Ситуация | Сообщение |
|---|---|
| Деление на ноль | `division by zero` |
| Выход за границы массива | `index out of bounds` |
| Вызов `panic` | передаётся аргумент функции |

Где `<N>` — номер строки исходного файла, в которой произошла ошибка.

Каждый исходный файл является модулем. Все объявления верхнего уровня файла образуют область видимости модуля. Между модулями нет явного механизма импорта в базовой версии языка — каждый файл компилируется и компонуется независимо.

## 12. Система эффектов

Функции могут быть аннотированы атрибутами эффектов. Это позволяет компилятору статически проверять, что функции не нарушают заявленных гарантий.

### 12.1 Распознаваемые атрибуты

| Атрибут | Значение |
|---------|----------|
| `@pure` | Функция не производит I/O и не может вызвать panic. |
| `@io` | Функция может читать/писать в стандартные потоки ввода/вывода. |
| `@panics` | Функция может вызвать panic (явно через `panic`, или неявно через деление на ноль / выход за границы массива). |

Функция без аннотаций эффектов **не проверяется** — она может делать что угодно. Это обеспечивает постепенное внедрение эффектов в проект.

### 12.2 Эффекты вызовов

Каждая операция в теле функции имеет **набор эффектов**:

| Операция | Эффекты |
|----------|---------|
| Арифметика, логика, сравнения, cast, доступ к полю/элементу (не целочисленное деление, не индексирование) | ∅ |
| Целочисленное деление `/`, остаток `%` (для `int8..int64`, `uint8..uint64`) | `{@panics}` (возможно деление на ноль) |
| Деление `/` для `float32`, `float64` | ∅ (IEEE 754: `±inf`, `NaN`, без runtime error) |
| Индексирование массива `a[i]` | `{@panics}` (возможен выход за границы) |
| Конкатенация строк, создание массива/структуры | ∅ |
| Чтение/запись переменной | ∅ |
| Создание диапазона `a..b` | ∅ |
| Вызов `print`, `println`, `input` | `{@io}` |
| Вызов `panic` | `{@panics}` |
| Вызов `exit` | `{@panics}` |
| Вызов `len` | ∅ |
| Вызов пользовательской функции `f` с аннотациями `{A1, ..., An}` | `{A1, ..., An}` |
| Вызов пользовательской функции **без** аннотаций | **неизвестные** |

### 12.3 Правило проверки `@pure`

Функция, объявленная как `@pure`, подчиняется следующему правилу:

> Набор эффектов её тела должен быть пустым.

Это означает, что в теле `@pure`-функции:
- нельзя вызывать `@io`, `@panics` функции;
- нельзя использовать **целочисленное** деление `/`, остаток `%` и индексирование `[]` — эти операции несут `@panics`;
- нельзя вызывать функции без аннотаций (их эффекты неизвестны).

**Допустимо** в теле `@pure`-функции:
- деление `/` для `float32`, `float64` — float-деление не несёт эффектов (см. §6.1, §12.2). Результаты `inf`/`NaN` не являются runtime error.

Исключение: `@pure` функция может вызывать **другую** `@pure` функцию.

Это строгое правило. Альтернатива — разрешить `@pure` + `@panics` вместе (т.е. `@pure` означает «только без I/O») — не принята намеренно: `@pure` в этом языке означает **тотальную** функцию (без эффектов, без runtime errors). Float-деление допускается потому, что в IEEE 754 оно тотально определено и не приводит к остановке программы.

**Следствие:** в `@pure` функции нельзя использовать целочисленные `/`, `%`, `a[i]` напрямую. Float-арифметика допустима полностью. В более зрелой версии языка здесь мог бы появиться отдельный эффект `@partial` или проверяемые компилятором гарантии для целочисленных операций.

### 12.4 Правила проверки `@io` и `@panics`

`@io` и `@panics` — **разрешающие** аннотации, не запрещающие. Функция с `@io` может делать I/O (но не обязана). Функция с `@panics` может паниковать.

Правило:

> Функция с набором аннотаций `A` должна содержать операции, эффекты которых — подмножество `A`.

Примеры:

```
fn foo() -> int32 @io {
    println("hi");      -- {@io} ⊆ {@io} ✓
    return 0;
}

fn bar() -> int32 @io {
    var x = 10 / 0;     -- {@panics} ⊄ {@io} — ошибка
    return x;
}

fn baz() -> int32 @io, @panics {
    var arr: array[int32, 3] = [1, 2, 3];
    println(arr[0]);    -- {@io, @panics} ⊆ {@io, @panics} ✓
    return 0;
}
```

### 12.5 Неаннотированные функции

Функция без аннотаций не проверяется сама и не может быть вызвана из `@pure`-функции (её эффекты неизвестны, а значит — потенциально непусты).

Она **может** быть вызвана из `@io`-функции или `@panics`-функции, но это считается «разрывом» эффектной системы. Компилятор выдаёт **предупреждение**:

```
warning: function 'foo' calls unannotated function 'bar';
         effect checking is incomplete at line <N>
```

Это предупреждение можно игнорировать — компиляция не прерывается. Цель — стимулировать постепенное внедрение аннотаций.

**Уточнения к правилу:**

а) **`main` исключена.** Если `main` объявлена без аннотаций (типичный случай — см. §12.6), вызовы неаннотированных функций из её тела **не** приводят к warning. Эффект-чек не запускается на неаннотированных функциях вообще, и `main` не является исключением из этого правила. Соответственно, программа, в которой и `main`, и вызываемые ею пользовательские функции — без аннотаций, компилируется молча.

б) **Один warning на объявление.** Для конкретной функции-вызывающего компилятор выдаёт **не более одного** warning о вызовах неаннотированных функций — на всё её тело, а не на каждый сайт вызова. Это исключает лавину одинаковых сообщений в крупных функциях, постепенно мигрирующих на эффекты. Локация warning'а — позиция первого встретившегося неаннотированного вызова.

в) **Избыточные эффекты в аннотации не диагностируются.** Функция, объявленная как `@io, @panics`, но фактически использующая только подмножество (например, только `@io`), **не** считается ошибкой и не порождает warning. Это отдельная тема (потенциальная будущая диагностика «declared effect E is unused»), и она вне рамок текущей версии эффектной системы. Декларация эффектов читается как «верхняя граница» допустимых операций тела.

### 12.6 Главная функция `main`

Для `main` аннотации эффектов не обязательны. Если они указаны — проверяются по общим правилам.

### 12.7 Эффекты методов

Правила для методов структур — ровно те же. Аннотация относится к методу в целом (включая неявный параметр `self`).

## 13. Перегрузка функций и разрешение вызовов

### 13.1 Объявление перегрузок

В одной области видимости (на уровне модуля, в пространстве имён, в блоке `impl`) может быть несколько функций с одним именем, если они различаются сигнатурой (§ `types.md`, §10).

При загрузке символов компилятор группирует все функции с одним именем в **множество перегрузок**. Каждый вызов имени разрешается по этому множеству.

### 13.2 Алгоритм разрешения вызова `f(a1, a2, ..., aN)`

1. Найти множество перегрузок `F` по имени `f`.
2. Если `F` пусто — ошибка `undefined function 'f'`.
3. Вычислить типы аргументов `T1, T2, ..., TN`.
4. Выделить из `F` подмножество **применимых** перегрузок `F_app`:
   - перегрузка `(P1, ..., PN) -> R` применима, если:
     - количество параметров совпадает с `N`;
     - каждый `Ti` совместим с `Pi` одним из трёх способов, каждому из которых присвоен **ранг** (меньше — точнее):
       - **ранг 0 — точное совпадение**: `Ti = Pi` (имена типов одинаковы);
       - **ранг 1 — подстройка литерала**: аргумент `ai` — безсуффиксный числовой литерал, `Pi` — числовой тип, и значение литерала входит в диапазон `Pi`;
       - **ранг 2 — неявное расширяющее приведение** (A.3.1): аргумент `ai` — выражение числового типа `Ti`, и существует **value-preserving** расширение `Ti → Pi`. Допустимые расширения: знаковое → знаковое большей разрядности; беззнаковое → беззнаковое большей разрядности; беззнаковое → знаковое большей разрядности; `float32 → float64`; любое целое → любое вещественное. Сужающие и лоссовые приведения (например, `int64 → int32`, `float64 → int32`, `float64 → float32`) неявно **не** выполняются — только через `cast<T>`.
5. Если `F_app` пусто — ошибка `no matching overload for call to 'f'` (с перечислением доступных перегрузок в диагностике).
6. **Выбор лучшей перегрузки.** Для каждой применимой перегрузки её **цена** — это сумма рангов её аргументов. Выбирается перегрузка с **минимальной** ценой.
   - если такая ровно одна — она выбирается;
   - если несколько перегрузок делят минимальную цену — ошибка `ambiguous call to 'f'` (с перечислением кандидатов).

Это правило обеспечивает:
- программист может использовать числовые литералы без суффиксов и они «найдут» нужную перегрузку, если та единственна;
- более конкретная (точная) перегрузка всегда предпочитается расширению: при наличии перегрузки с ценой 0 перегрузки с расширениями (цена ≥ 2) не выигрывают;
- если перегрузки конфликтуют (равная цена) — например, безсуффиксный литерал подходит двум целым типам, или один аргумент расширяется до двух разных типов одинаково — программист должен явно указать тип (`cast<int32>(5)` или суффикс `5L`).

> **Замечание (A.3.1).** Неявные расширяющие приведения действуют **только** в контексте выбора перегрузки. Во всех остальных местах (присваивание, инициализация, операнды бинарных операций) общий запрет неявных приведений сохраняется — там по-прежнему требуется явный `cast<T>`.

### 13.3 Примеры разрешения

```
fn foo(x: int32) -> int32 @pure { return x; }
fn foo(x: int64) -> int64 @pure { return x; }

fn main() -> int32 {
    foo(5);              -- неоднозначность: 5 может быть int32 и int64
                         -- ошибка: ambiguous call
    foo(cast<int32>(5)); -- int32 — точное совпадение
    foo(5L);             -- int64 — точное совпадение
    return 0;
}
```

```
fn bar(x: int32) -> int32 @pure { return x; }
fn bar(x: int32, y: int32) -> int32 @pure { return x + y; }

fn main() -> int32 {
    bar(5);              -- одна подходит по количеству аргументов
    bar(5, 10);          -- другая
    return 0;
}
```

Неявное расширение при разрешении перегрузки (A.3.1):

```
fn twice(x: int64) -> int64 @pure { return x + x; }

fn pick(x: int32) -> int32 @pure { return 1; }
fn pick(x: int64) -> int64 @pure { return 2; }

fn main() -> int32 {
    var a: int32 = 21;
    twice(a);            -- единственная перегрузка: int32 расширяется до int64 (цена 2)
    pick(a);             -- pick(int32) — точное совпадение (цена 0) предпочитается
                         -- расширению до pick(int64) (цена 2): выбирается pick(int32)
    return 0;
}
```

### 13.4 Разрешение методов

Для вызова `obj.method(args)`:
1. Тип `obj` определяет пространство имён методов.
2. Множество перегрузок `method` строится в этом пространстве.
3. Первый параметр (`self: T`) сопоставляется с `obj`; далее — стандартный алгоритм §13.2.

Для `T::name(args)` — аналогично, но без параметра `self`.

### 13.5 Вызов перегруженной функции из перегруженной

Рекурсивные вызовы внутри перегрузки разрешаются как обычные вызовы — с полным множеством перегрузок. Это позволяет одной перегрузке вызвать другую:

```
fn sum(a: int32, b: int32) -> int32 @pure {
    return a + b;
}

fn sum(a: int32, b: int32, c: int32) -> int32 @pure {
    return sum(a, b) + c;   -- вызывает первую перегрузку
}
```

## 14. Обработка ошибок синтаксического анализа

Парсер реализует **восстановление после синтаксических ошибок** по схеме **panic mode**:

### 14.1 Основные принципы

1. При обнаружении синтаксической ошибки парсер:
   - выдаёт диагностическое сообщение;
   - **не прекращает работу**, а пытается восстановиться;
   - продолжает поиск ошибок на оставшейся части файла.
2. Компиляция прекращается **после фазы парсинга**, если была хотя бы одна ошибка — семантический анализатор не запускается.
3. Код возврата компилятора — ненулевой при наличии хотя бы одной ошибки.

### 14.2 Точки синхронизации

Парсер пропускает токены до ближайшей **точки синхронизации** и продолжает разбор с неё. Точки синхронизации:

- `;` — конец инструкции (пропустить сам `;`, продолжить со следующего токена);
- `}` — конец блока (остановиться **перед** `}`, дать внешнему правилу его обработать);
- ключевые слова, начинающие инструкции верхнего уровня или объявления: `fn`, `struct`, `namespace`, `impl`, `type`, `var`, `const`, `if`, `while`, `for`, `return`.

### 14.3 Ограничение количества ошибок

Парсер собирает ошибки и в конце выводит их все. Если количество накопленных ошибок превышает определённый порог (реализация рекомендует ≥ 20), парсер останавливает выдачу новых сообщений:

```
... 20 more errors suppressed
```

Это предотвращает «взрывной» каскад ошибок из-за сбитой синхронизации.

### 14.4 Вложенные ошибки

После восстановления некоторые последующие «ошибки» могут быть артефактами неверной синхронизации, а не реальными проблемами в коде. Компилятор **не различает** настоящие и артефактные ошибки — программист видит все и исправляет по одной, перекомпилируя.

## 15. Промежуточное представление и оптимизации

Между семантическим анализатором и генератором кода вводится фаза линеаризации: аннотированный AST преобразуется в **линейное промежуточное представление (IR)** в форме трёхадресного кода (TAC).

Подробное описание инструкций IR, формата и правил преобразования — в `codegen.md`.

На уровне IR реализуются следующие оптимизации:

### 15.1 Constant folding

Арифметические, логические операции и сравнения с двумя константными операндами вычисляются на этапе компиляции:

```
-- до:
t1 = 3 + 4
t2 = t1 * 2

-- после constant folding:
t1 = 7
t2 = 14
```

Применяется многократно до достижения фиксированной точки.

Ограничения:
- Не производится для операций, которые могут вызвать runtime error с известными операндами (деление на известный ноль не сворачивается в ошибку — оставляется для run-time).
- Не производится для `float` операций, если результат может зависеть от режима округления (не актуально для базовой реализации, но зарезервировано).

### 15.2 Dead code elimination

После constant folding запускается проход по IR, удаляющий:

- присваивания временным переменным, которые далее нигде не используются;
- блоки IR, недостижимые по потоку управления (например, код после `return`, `panic`, `exit`);
- ветки `if` с константным условием — пропадает одна из веток;
- циклы `while false { ... }` — всё тело удаляется.

Результат работы оптимизатора — IR, семантически эквивалентный исходному, но меньшего размера.

## 16. Семантика на границах фаз

### 16.1 Гарантии после семантического анализа

По окончании работы семантического анализатора аннотированный AST удовлетворяет следующим инвариантам:

- каждый идентификатор разрешён (имеет ссылку на объявление);
- каждое выражение имеет точный известный тип;
- каждый вызов функции разрешён в конкретную перегрузку;
- каждый `break`/`continue` ссылается на конкретный цикл;
- для каждого `return` известна функция, из которой он выходит;
- все аннотации эффектов проверены;
- все перемещения `range[T]` проверены на отсутствие повторного использования;
- все `defer` связаны со своими блоками выхода.

Фаза lowering в IR полагается на эти инварианты и их не перепроверяет.

### 16.2 Порядок проверок

Семантический анализатор выполняет проверки в следующем порядке:

1. Сбор объявлений всех верхнеуровневых сущностей (функции, структуры, `impl`-блоки, синонимы, пространства имён).
2. Проверка корректности типов в объявлениях (поля структур, параметры функций, синонимы).
3. Проверка групп перегрузок на отсутствие дубликатов сигнатур.
4. Обход тел функций — разрешение имён, проверка типов, вывод типов, разрешение перегрузок.
5. Проверка управления потоком — `break`/`continue` внутри циклов, корректность `return`.
6. Проверка move-семантики для `range[T]`.
7. Проверка эффектов.

Ошибка на любой стадии не прекращает анализ (аналогично парсеру — накопление ошибок), но проваливает компиляцию.
