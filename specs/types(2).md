# types.md — Система типов

---

## 1. Общие принципы

Язык использует **статическую** и **номинальную** систему типов:

- **Статическая** — тип каждого выражения определяется на этапе компиляции.
- **Номинальная** — два типа совместимы только если их имена совпадают. Структурное сходство не делает типы совместимыми.
- **Отсутствие неявных приведений** — любое преобразование типов должно быть явным через `cast<T>(expr)`.
- **Value semantics** — все значения передаются и присваиваются по значению (копированием). Исключение — тип `range[T]` (см. §8).
- **Локальный вывод типов** — тип переменной в `var`/`const` может быть опущен и выведен из инициализатора.

## 2. Встроенные типы

### 2.1 Целочисленные типы

| Тип | Размер | Диапазон |
|------|--------|----------|
| `int8` | 8 бит | −128 .. 127 |
| `int16` | 16 бит | −32 768 .. 32 767 |
| `int32` | 32 бит | −2 147 483 648 .. 2 147 483 647 |
| `int64` | 64 бит | −2⁶³ .. 2⁶³ − 1 |
| `uint8` | 8 бит | 0 .. 255 |
| `uint16` | 16 бит | 0 .. 65 535 |
| `uint32` | 32 бит | 0 .. 4 294 967 295 |
| `uint64` | 64 бит | 0 .. 2⁶⁴ − 1 |

### 2.2 Вещественные типы

| Тип | Размер | Стандарт |
|------|--------|----------|
| `float32` | 32 бит | IEEE 754 single precision |
| `float64` | 64 бит | IEEE 754 double precision |

### 2.3 Булев тип

| Тип | Значения |
|------|----------|
| `bool` | `true`, `false` |

`bool` не является числовым типом. Неявное преобразование между `bool` и целыми числами не допускается.

### 2.4 Строковый тип

| Тип | Описание |
|------|----------|
| `string` | Последовательность байт в кодировке UTF-8, неизменяемая |

Строки передаются по значению (копирование). Размер строки не фиксирован и определяется во время выполнения.

### 2.5 Тип hollow

| Тип | Описание |
|------|----------|
| `hollow` | Отсутствие значения. Используется как тип возврата функций без результата и как тип блока, не завершающегося выражением |

Переменная типа `hollow` не может быть объявлена. `hollow` допустим как тип возврата функции и как тип блочного выражения, значение которого не используется.

### 2.6 Массивы

```
array[T, N]
```

- `T` — тип элементов (любой тип кроме `hollow` и `range[...]`)
- `N` — размер, целочисленный литерал > 0, фиксирован на этапе компиляции
- Размер является **частью типа**: `array[int32, 3]` и `array[int32, 5]` — разные типы
- Элементы индексируются с `0` до `N − 1`
- Массив передаётся по значению (полное копирование)

### 2.7 Структуры

Объявляются через `struct`. Два типа-структуры совместимы только если у них совпадают имена (номинальная типизация). Структуры с одинаковым набором полей, но разными именами — разные типы.

Методы и ассоциированные функции структуры объявляются в блоках `impl` (см. §9).

### 2.8 Диапазоны — `range[T]`

```
range[T]
```

- `T` — целочисленный тип (`int8`..`int64`, `uint8`..`uint64`)
- Значения типа `range[T]` порождаются выражением `a..b`, где `a` и `b` имеют тип `T`
- Диапазон представляет **полуинтервал** `[a, b)`: включает `a`, не включает `b`
- Если `b < a` (для знаковых) или `b <= a` (в любом случае) — диапазон пуст

**Особенности типа `range[T]`:**

Диапазон не подчиняется обычной value semantics языка — он не копируется, а **перемещается** при присваивании и передаче в функцию. Подробности — §8 и `semantics.md`, §7.5.

Допустимые места использования `range[T]`:
- как тип итерируемого выражения в `for`-цикле;
- как тип параметра функции;
- как тип локальной переменной (с учётом ограничений §8);
- как тип возвращаемого значения функции.

Диапазон **не может** быть:
- полем структуры;
- элементом массива;
- операндом `cast<T>`;
- сравниваться, передаваться в `print`.

## 3. Пользовательские типы

### 3.1 Синонимы типов

```
type Meters = int32;
```

Синоним **не создаёт новый тип** — это альтернативное имя для существующего. Значения типа `Meters` и `int32` полностью взаимозаменяемы без приведения. Синоним можно создать для любого типа, включая `array[...]` и `range[...]`.

### 3.2 Структуры

Структура создаёт новый именованный тип. Поля структуры имеют определённый порядок. Доступ к полям — через оператор `.`, к методам — через `.method(...)`, к ассоциированным функциям — через `Type::function(...)`.

## 4. Типы литералов

| Литерал | Тип по умолчанию |
|---------|-----------------|
| `42` | `int32` |
| `42u` | `uint32` |
| `42L` | `int64` |
| `42uL` | `uint64` |
| `3.14` | `float64` |
| `3.14f` | `float32` |
| `true` / `false` | `bool` |
| `"..."` | `string` |
| `[1, 2, 3]` | `array[int32, 3]` (тип элемента выводится из первого элемента) |

Если числовой литерал без суффикса используется в контексте, где ожидается конкретный числовой тип (правая часть `var x: int16 = 5`, аргумент функции с параметром типа `int16`, и т.п.), литерал интерпретируется как значение этого типа при условии, что оно входит в допустимый диапазон. Иначе — ошибка компиляции.

Это свойство распространяется **только на безсуффиксные литералы**. Литерал `5L` имеет жёсткий тип `int64` и не может быть использован там, где ожидается `int32`, без явного `cast`.

## 5. Вывод типов

Язык поддерживает **локальный вывод типов** для объявлений `var` и `const`.

### 5.1 Вывод типа переменной

Если в `var` или `const` опущена аннотация типа, тип переменной выводится из типа инициализирующего выражения:

```
var a = 10;              -- a : int32
var b = 3.14;            -- b : float64
const name = "Kostya";   -- name : string
var flag = true;         -- flag : bool
var arr = [1, 2, 3];     -- arr : array[int32, 3]
```

Тип инициализатора определяется **без учёта** ожидаемого типа переменной — так как последнего просто нет. В частности, безсуффиксный числовой литерал в отсутствие аннотации получает свой тип **по умолчанию** (`int32` для целых, `float64` для вещественных).

```
var x = 5;               -- x : int32  (не int16, даже если позже используется как int16)
var y: int16 = 5;        -- y : int16  (литерал подстраивается под аннотацию)
```

### 5.2 Ограничения вывода

Вывод типа невозможен, если тип инициализатора не определён однозначно:

- Инициализатор `[]` (пустой массив) — тип элемента не выводится, требуется аннотация.
- Инициализатор `struct_name { ... }` с элементами смешанных числовых литералов — обрабатывается стандартно, вывод идёт по объявленным типам полей структуры.

В случаях, когда вывод невозможен, компилятор выдаёт ошибку с сообщением `cannot infer type of <name>`.

### 5.3 Отсутствие глобального вывода

Вывод типов не работает на следующих позициях — там тип обязан быть указан явно:

- Параметры функций;
- Тип возвращаемого значения функции;
- Поля структур;
- Синонимы типов (`type Meters = int32;` — правая часть всегда тип, а не выражение).

## 6. Совместимость типов

Два типа **совместимы** (могут использоваться в одном выражении без приведения) только если они идентичны по имени.

Примеры несовместимых пар:

```
int32  ≠  int64     -- разные типы, нужен cast
int32  ≠  uint32    -- знаковый ≠ беззнаковый
float32 ≠ float64
bool   ≠  int32
array[int32, 3] ≠ array[int32, 5]   -- разный размер
range[int32]    ≠ range[int64]      -- разный тип элемента
Point  ≠  Point2D                   -- разные имена структур
```

## 7. Явное приведение типов

Синтаксис:

```
cast<T>(expr)
```

### 7.1 Допустимые приведения

#### Числовые приведения

Допускаются приведения между всеми числовыми типами (`int8`..`int64`, `uint8`..`uint64`, `float32`, `float64`):

```
var x: int32 = 1000;
var y: float64 = cast<float64>(x);
var z: int16  = cast<int16>(x);
```

#### bool ↔ int

```
cast<int32>(true)   -- результат: 1
cast<int32>(false)  -- результат: 0
cast<bool>(1)       -- результат: true  (любое ненулевое значение)
cast<bool>(0)       -- результат: false
```

#### Недопустимые приведения

Следующие приведения являются ошибкой компиляции:

| От | К | Причина |
|----|---|---------|
| `string` | любой числовой | не определено |
| любой числовой | `string` | не определено |
| `struct` | любой другой | не определено |
| `array` | любой другой | не определено |
| `range` | любой другой | не определено |
| любой | `hollow` | бессмысленно |
| любой | `range[...]` | не определено |

### 7.2 Поведение при narrowing

При приведении к типу с меньшим диапазоном (**narrowing**) лишние биты отбрасываются (усечение):

```
var x: int32 = 300;
var y: int8  = cast<int8>(x);   -- y == 44  (300 mod 256)
```

Это поведение определено и не является ошибкой. Ответственность за корректность лежит на программисте.

### 7.3 Поведение при приведении float → int

Дробная часть отбрасывается (усечение к нулю):

```
cast<int32>(3.9)    -- результат: 3
cast<int32>(-3.9)   -- результат: -3
```

## 8. Не-копируемые типы

Тип `range[T]` является **не-копируемым** (non-copyable). Это единственное исключение из общей value semantics языка.

### 8.1 Семантика перемещения

При операциях, которые в обычной value semantics вызвали бы копирование — присваивании, передаче в функцию, возврате из функции, использовании в качестве инициализатора — значение типа `range[T]` **перемещается**: после операции источник становится **потреблённым** и не может использоваться повторно. Повторное использование — ошибка компиляции:

```
var r = 0..10;
var s = r;          -- r потреблён
for x in r { ... }  -- ошибка компиляции: r уже потреблён
```

### 8.2 Потребление в `for`-цикле

Использование диапазона в `for`-цикле **потребляет** его:

```
var r = 0..5;
for i in r { print(i); }
for i in r { print(i); }    -- ошибка компиляции: r потреблён
```

Диапазон-литерал, переданный в `for` напрямую, не требует промежуточной переменной и проблем не создаёт:

```
for i in 0..5 { print(i); }
for i in 0..5 { print(i); }   -- корректно: два разных временных значения
```

### 8.3 Обоснование

Диапазоны имеют внутреннее состояние итерации. Копирование такого значения привело бы к неочевидным результатам (две копии итерируют независимо? синхронно?). Выбор в пользу move-семантики упрощает как спецификацию, так и реализацию.

Формальные правила перемещения и проверки — см. `semantics.md`, §7.5.

## 9. Методы и ассоциированные функции

Для каждой структуры может быть объявлен один или несколько блоков `impl`. Все объявления из всех блоков `impl` данного типа объединяются в единое пространство имён методов и ассоциированных функций этого типа.

### 9.1 Методы

Метод — функция в блоке `impl StructName`, первый параметр которой объявлен как `self: StructName`:

```
impl Point {
    fn distance(self: Point, other: Point) -> float64 @pure { ... }
}
```

Вызов метода:

```
var p: Point = ...;
var q: Point = ...;
var d = p.distance(q);
```

Выражение `p.distance(q)` эквивалентно вызову функции с первым аргументом `p`:

```
Point::distance(p, q)
```

Обе формы допустимы одновременно.

Внутри тела метода доступно ключевое слово `self`, имеющее тип структуры:

```
impl Point {
    fn norm(self: Point) -> float64 @pure {
        return self.x * self.x + self.y * self.y;
    }
}
```

### 9.2 Ассоциированные функции

Функция без параметра `self` в блоке `impl StructName` называется **ассоциированной функцией** типа. Она не имеет получателя и вызывается через `::`:

```
impl Point {
    fn origin() -> Point @pure {
        return Point{ x: 0.0, y: 0.0 };
    }

    fn make(x: float64, y: float64) -> Point @pure {
        return Point{ x: x, y: y };
    }
}

var o = Point::origin();
var p = Point::make(3.0, 4.0);
```

Ассоциированные функции — стандартный способ объявить конструкторы и утилитарные функции, логически привязанные к типу.

### 9.3 Разрешение имён методов

Имена методов и ассоциированных функций принадлежат **пространству имён типа**, а не обычной области видимости. Поэтому допустимо существование обычной функции `distance` и метода `distance` у `Point` одновременно — они не конфликтуют.

Внутри блоков `impl` допускаются **перегрузки** по общим правилам (§10): несколько методов с одним именем, различающихся количеством или типами параметров.

### 9.4 Ограничения

- `impl` допустим только для типов-структур, объявленных в текущем модуле или в пространстве имён, внутри которого расположен `impl`.
- Нельзя объявить `impl` для встроенного типа (`int32`, `string` и т.д.) или синонима типа.
- Имя в `self: T` должно совпадать с именем структуры в заголовке `impl`.

## 10. Перегрузка функций

В одной области видимости допускается несколько объявлений функции с одним именем, если они различаются **сигнатурой параметров** — количеством параметров и/или их типами.

### 10.1 Что входит в сигнатуру

Сигнатура для целей перегрузки:
- имя функции;
- количество параметров;
- типы параметров в порядке объявления.

В сигнатуру **не входят**:
- имена параметров;
- тип возвращаемого значения;
- аннотации эффектов.

Следующие пары — корректные перегрузки:

```
fn area(side: float64) -> float64 @pure;
fn area(w: float64, h: float64) -> float64 @pure;
```

Следующие пары — ошибка компиляции (совпадающие сигнатуры):

```
fn foo(x: int32) -> int32;
fn foo(x: int32) -> float64;            -- отличается только тип возврата

fn bar(a: int32) -> int32;
fn bar(b: int32) -> int32;              -- отличается только имя параметра

fn baz(x: int32) -> int32 @pure;
fn baz(x: int32) -> int32 @io;          -- отличается только эффект
```

### 10.2 Разрешение перегрузок

При вызове функции компилятор выбирает единственную перегрузку, в которой **типы всех аргументов точно совпадают** с типами параметров. Правила разрешения детально описаны в `semantics.md`, §13. Ключевые моменты:

- совпадение должно быть точным — неявных приведений нет;
- безсуффиксные числовые литералы могут «подстраиваться» под требуемый тип (см. §4), но если несколько перегрузок допускают такую подстройку, это неоднозначность — ошибка;
- если не найдена ни одна подходящая перегрузка — ошибка `no matching overload`;
- если подходит более одной — ошибка `ambiguous call`.

### 10.3 Перегрузка между функциями и методами

Обычные функции и методы структур находятся в **разных пространствах имён** и между собой не перегружают друг друга. Метод `Point::make` и функция `make` могут сосуществовать без взаимного конфликта.

## 11. Эффекты в сигнатурах функций

Каждая функция (включая методы) может быть аннотирована одним или несколькими атрибутами эффектов:

```
fn foo() -> int32 @pure { ... }
fn bar() -> hollow @io { ... }
fn baz(x: int32) -> int32 @io, @panics { ... }
```

### 11.1 Распознаваемые атрибуты

| Атрибут | Смысл |
|---------|-------|
| `@pure` | Функция не производит I/O и не может вызвать `panic`/`exit`. |
| `@io` | Функция может читать/писать в стандартные потоки. |
| `@panics` | Функция может вызвать `panic` (в т.ч. неявно — через деление на ноль, выход за границы массива). |

Функция, не имеющая аннотаций, считается **нетипизированной по эффектам** — к ней не применяются ограничения эффектной системы. Это сделано для обратной совместимости и постепенного внедрения: можно аннотировать только отдельные функции.

Совместимость атрибутов и правила проверки — `semantics.md`, §12.

### 11.2 Эффекты в перегрузках и подстановке типов

Эффекты **не входят в сигнатуру** для разрешения перегрузок. Изменение только эффекта — не перегрузка, а конфликт.

Эффекты **не являются частью типа функции** в том смысле, что нет «типа функции-значения» в языке (функции не являются значениями первого класса). Эффекты — это отдельный атрибут, проверяемый на каждом вызове.

## 12. Проверка типов в выражениях

### 12.1 Унарные операторы

| Оператор | Допустимые типы операнда | Тип результата |
|----------|--------------------------|----------------|
| `-` | числовые | тот же тип |
| `!` | `bool` | `bool` |

### 12.2 Бинарные арифметические операторы

| Оператор | Допустимые типы | Тип результата |
|----------|----------------|----------------|
| `+` `-` `*` `/` | числовые (оба одного типа) | тот же тип |
| `%` | целочисленные (оба одного типа) | тот же тип |
| `+` | `string` | `string` |

### 12.3 Операторы сравнения

| Оператор | Допустимые типы | Тип результата |
|----------|----------------|----------------|
| `==` `!=` | числовые, `bool`, `string` (оба одного типа) | `bool` |
| `<` `>` `<=` `>=` | числовые (оба одного типа) | `bool` |

### 12.4 Логические операторы

| Оператор | Допустимые типы | Тип результата |
|----------|----------------|----------------|
| `&&` `\|\|` `!` | `bool` | `bool` |

### 12.5 Оператор диапазона `..`

| Оператор | Допустимые типы | Тип результата |
|----------|----------------|----------------|
| `..` | целочисленные (оба одного типа) | `range[T]` |

### 12.6 Оператор `|>` (pipe)

`expr |> callable` имеет тип возвращаемого значения вызова `callable(expr, ...)`. Проверка:
- правая часть — имя функции, `Type::fn` или вызов функции `f(args)`;
- если правая часть — имя, то функция должна иметь ровно один параметр, тип которого совпадает с типом `expr`;
- если правая часть — вызов `f(args)`, то тип `expr` должен совпадать с типом **первого** параметра `f`, а `args` — с типами остальных параметров.

Результат `a |> f(b, c)` имеет тип, равный типу возврата `f`.

### 12.7 Условное выражение `if`

`if_expr` имеет тип, равный общему типу обеих веток:
- типы веток должны быть идентичны;
- если типы различаются — ошибка компиляции.

### 12.8 Блочное выражение

Тип `block_expr`:
- если блок завершается выражением без `;` — тип этого выражения;
- иначе — `hollow`.

---

## 13. Типы в инструкциях управления

### 13.1 `if`/`while` (как инструкции)

Условие обязано иметь тип `bool`. Ветки — блочные выражения, значение которых игнорируется.

### 13.2 `for`

Итерируемое выражение обязано иметь тип `range[T]` или `array[T, N]`. Тип переменной цикла:
- для `range[T]` — `T`;
- для `array[T, N]` — `T`.

Переменная цикла иммутабельна в теле.

### 13.3 `return`

В функции с типом возврата `R ≠ hollow`: тип выражения после `return` должен совпадать с `R`.

В функции с типом возврата `hollow`: `return` без выражения.

---

## 14. Типы параметров и возврата функций

Параметры функции могут быть любого типа кроме `hollow`. Тип `range[T]` допустим, но подчиняется move-семантике (§8).

Тип возврата может быть любым типом включая `hollow` и `range[T]`.

При вызове функции тип каждого аргумента должен точно совпадать с типом соответствующего параметра. Несоответствие — ошибка компиляции (с учётом подстройки безсуффиксных числовых литералов, §4).

---

## 15. Типизация структурных литералов

При создании литерала структуры:

```
var p: Point = Point{ x: 3.0, y: 4.0 };
```

- Все поля должны быть перечислены.
- Порядок полей не важен — поля идентифицируются по имени.
- Тип каждого значения поля должен совпадать с объявленным типом поля (с учётом подстройки числовых литералов).
- Лишние или отсутствующие поля — ошибка компиляции.

---

## 16. Типизация массивных литералов

```
var nums: array[int32, 3] = [1, 2, 3];
```

- Все элементы литерала должны быть одного типа.
- Количество элементов должно совпадать с `N` в типе переменной.
- Несоответствие количества или типа элементов — ошибка компиляции.

При использовании литерала в позиции с опущенной аннотацией (`var nums = [1, 2, 3];`) тип определяется по первому элементу, размер — по количеству элементов. Если элементы имеют разные типы — ошибка компиляции.

---

## 17. Операции над строками

| Операция | Синтаксис | Тип результата |
|----------|-----------|----------------|
| Конкатенация | `s1 + s2` | `string` |
| Равенство | `s1 == s2` | `bool` |
| Неравенство | `s1 != s2` | `bool` |
| Длина | `len(s)` | `int32` |

`len` — встроенная функция, применимая только к `string`. Возвращает количество **байт** (не символов UTF-8).

Другие операторы сравнения (`<`, `>`, `<=`, `>=`) для `string` не определены и являются ошибкой компиляции.

---

## 18. Переполнение

Переполнение целочисленных типов ведёт себя как **wraparound** (беззнаковая арифметика по модулю 2ⁿ для `uint`, знаковое переполнение для `int` определено как wraparound).

Переполнение **не является** ошибкой времени выполнения — поведение предсказуемо.

```
var x: uint8 = 255;
x = x + cast<uint8>(1);   -- x == 0  (wraparound)

var y: int8 = 127;
y = y + cast<int8>(1);    -- y == -128  (wraparound)
```

Для вещественных типов: поведение соответствует стандарту IEEE 754 (бесконечность, NaN).
