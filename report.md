# Отчёт о работе

Компилятор языка L (`myc`): `.lang` - нативный ELF. Фронтенд и middle-end
написаны на C++23 (модули, без исключений и RTTI), бэкенд — генерация
LLVM IR, дальше `llc` + системный линкер. 

## Архитектура

Пайплайн линейный, каждая фаза — отдельная статическая библиотека (`src/`):

исходник
| → lexer    (lexer/)    поток токенов
| → parser   (parser/)   AST
| → sema     (sema/)     проверки + аннотации, таблица типов
| → lowering (ir/)       трёхадресный IR
| → opt      (ir/opt/)   свёртка констант, DCE (по флагу -O)
| → codegen  (codegen/)  LLVM IR (текст)
| → llc + cc                       .o → исполняемый файл

Диагностики собираются в `diag/` через общий `DiagnosticEngine`; ошибки
не бросаются, а возвращаются (`std::expected` / коды), накапливаются и
печатаются в конце фазы.

Драйвер — `src/main.cpp`. Рантайм (`mycrt`, печать/паника/проверки) —
`runtime/runtime.c`.

### Принятые решения и их обоснование

- **Парсер — рекурсивный спуск.** Грамматика небольшая, спуск читается
 один в один со спецификацией грамматики;
- **Бэкенд — LLVM IR, а не свой кодоген x86-64.** Снимает с проекта
  распределение регистров и ABI, даёт оптимизации и переносимость почти
  бесплатно. Платим зависимостью от `llc` в рантайме сборки.
- **Отдельная фаза IR между sema и кодогеном.** Кодоген работает с
  плоским трёхадресным кодом, а не с AST; на IR живут оптимизации и
  отладочная печать (`--dump-ir`).
- **Интернирование типов.** Каждый тип существует в одном экземпляре,
  адресуется `TypeId`, совместимость — сравнение Id, без deep-compare.
- **Две семантические фазы.** Pass 1 собирает объявления верхнего уровня
  (forward-видимость), pass 2 проверяет тела.

## Сборка и запуск

Нужны g++ ≥ 14 (рекомендуется g++-15, из-за C++20-модулей `.cppm`),
CMake ≥ 3.28, генератор Ninja, LLVM (`llc`).

```bash
CXX=g++-15 cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build build -j$(nproc)
./build/src/myc --version
```

Тесты:

```bash
ctest --test-dir build          # юнит-тесты
src/tests/run_tests.sh          # golden + корпус .lang
```

Использование:

```bash
myc prog.lang                   # → исполняемый файл
myc prog.lang -o prog
myc --emit-ll prog.lang         # LLVM IR в файл
myc --emit-obj prog.lang        # объектный файл
myc --dump-tokens|--dump-ast|--dump-ir|--dump-llvm-ir prog.lang
myc -O0 / -O2 prog.lang         # оптимизации (по умолчанию включены)
```

`llc` берётся из `PATH` или из переменной `MYC_LLC`.

## Реализованные дополнительные задания

Нумерация по `project v2.0.md`. (В `PROGRESS.md` коды идут по старой
редакции v1.0 и не совпадают с приведёнными ниже; char там не отражён.)

Полностью реализованы:

- **A.1.1** Числовые типы разных размеров — `int8..int64`, `uint8..uint64`,
  `float32/64`, суффиксы литералов (`u`, `L`, `uL`, `f`).
- **A.1.2** Unicode-символы — тип `char` как 32-битная кодовая точка,
  `code()` / `char_from()`, отдельный нечисловой тип.
- **A.1.11** Конвейерный оператор `|>` — дешугар в `f(x)`, частичное
  применение `x |> f(b, c)`. Композиция `>>` не делалась (опциональна).
- **A.2.5** Defer — LIFO-порядок, срабатывает на нормальном выходе,
  `return` и панике.
- **A.2.8** Перегрузка функций — выбор по числу и типам параметров,
  диагностики "no matching overload" / "ambiguous call".
- **A.3.1** Перегрузка с неявными расширяющими приведениями — ранги
  (точное < подстройка литерала < расширение), выбор минимальной цены,
  ничья → ошибка неоднозначности. *Расширяет A.2.8.*
- **B.2.1** Промежуточное представление (IR) — трёхадресный код,
  отдельная фаза lowering, печать `--dump-ir`, golden-тесты.
- **B.2.2** Интернирование типов — `TypeId`, сравнение по Id.
- **B.1.10** Golden-тесты — раннер `run_tests.sh` / `run_tests.py`,
  сверка stdout/stderr/кода возврата, есть и негативные программы.

Реализованы частично (по правилам ТЗ к зачёту допа не предъявляются):

- **A.1.7** Автовывод типов — только локальный, для `var`/`const` по
  инициализатору; вывод типа возврата функций не сделан.
- **A.2.1** Expression-oriented — значение возвращают `if`- и блочные
  выражения; помеченных блоков с `break <значение>` нет.
- **A.2.12** Методы структур — `impl`-блоки, `self`, ассоциированные
  функции (`Type::fn`), UFCS, перегрузки в `impl`; уровни доступа
  `public`/`private` не реализованы.

Сверх списка A/B (в спецификации, баллов по рубрике не дают):

- эффекты `@pure` / `@io` / `@panics` с проверкой в sema;
- оптимизации на уровне IR (свёртка констант, DCE);
- move-семантика для `range[T]`;
- namespaces;
- runtime-проверки (выход за границы, деление на ноль) и `panic`.

## Известные ограничения

Расхождения со спецификацией, выявленные тест-корпусом, не исправлены:

- `self` вне тела метода роняет компилятор (assert вместо ошибки).
- `var x: hollow = {};` и type-alias как тип поля/параметра — падение на
  этапе `llc` («Cannot allocate unsized type»).
- Type alias внутри namespace, использованный как тип, — assert в sema.
- `range[T]` как параметр функции с итерацией — отказ `llc`.
- Голый доступ `Ns::CONST` / `Ns::VAR` снаружи namespace резолвится в
  `hollow`; снаружи работают только `Ns::fn()` и `Ns::Alias`.
- `nan != nan` даёт `false` (ordered fcmp вместо unordered), NaN
  печатается как `-nan`.
- Не отвергаются: присваивание элементу const-массива (`a[0] = 9`),
  присваивание параметру функции, литерал вне диапазона типа
  (`var x: int8 = 300`).
- `continue` в `for … in range` минует инкремент → зацикливание;
  затенение имён (два `var x` в разных блоках) ломает кодоген.

Не реализовано: указатели, generics, замыкания, файловые модули,
блочные комментарии, битовые операции, FFI.
