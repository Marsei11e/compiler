# Тест-сьют

Корпус из 126 программ на разрабатываемом языке для проверки компилятора
`myc`. Каждая программа — самодостаточный файл `.lang` с метаданными в
комментариях, описывающими ожидаемое поведение.

MYC=$PWD/build/src/myc src/tests/run_tests.sh -v
build/src/myc examples/99_integration/stats.lang -o stats

if-выражения, cast, struct-литералы, индексацию и доступ к namespace.

## Формат метаданных

Каждый файл начинается с комментариев вида `-- key: value` или
`-- key:\n-- ...`. Распознаются:

| Ключ | Тип | Значение |
|------|-----|----------|
| `expect_stdout` | многострочный | Точный ожидаемый stdout (сравнение построчно). Каждая строка после ключа до пустой строки или следующего ключа считается частью ожидаемого вывода. |
| `expect_exit` | целое | Код возврата процесса. По умолчанию `0`. |
| `expect_stderr_contains` | строка | Подстрока, которая должна присутствовать в stderr. Используется для runtime-ошибок и ошибок компиляции. |
| `expect_compile_error` | строка | Подстрока, которую должен содержать stderr **компилятора**. Программа не запускается. Если эта метка стоит — `expect_stdout` и `expect_exit` игнорируются. |
| `expect_runtime_error` | bool | (Альтернатива `expect_stderr_contains`.) Программа должна **скомпилироваться** успешно, но **упасть в runtime** (exit ≠ 0, stderr содержит `runtime error:`). |

### Как ранер парсит метаданные

Только **сплошной блок комментариев в начале файла** (до первой строки кода)
считается заголовком. Внутри заголовка строки вида `-- key: value`
или `-- key:` (с продолжением `-- v1`, `-- v2`, ...) — это метаданные.
Любая строка комментария, не похожая на метаданные (просто описание),
игнорируется парсером.

### Пример

```
-- 01_hello.lang
-- Минимальная программа.
-- expect_stdout:
-- Hello, world!
-- expect_exit:   0

fn main() -> int32 {
    println("Hello, world!");
    return 0;
}
```

`expect_stdout` — многострочный, начинается со строки после ключа и
включает все последующие `-- ...`-строки до пустой строки или следующей
метаданной. Здесь это одна строка `Hello, world!`.

## Запуск

```bash
./run_tests.sh                       # все тесты
./run_tests.sh 01_basic              # одна категория
./run_tests.sh 10_negative/07*       # один тест по glob
./run_tests.sh -v                    # подробный вывод
```

Скрипт `run_tests.sh` (см. рядом) проходит по всем `.lang`-файлам, для
каждого парсит метаданные, запускает компилятор и сравнивает результат.

Возвращает код 0 при полном успехе, 1 если хоть один тест провалился.

## Зависимости

- собранный `myc` в `$PATH` или указанный через `MYC=/path/to/myc`;
- `bash`, `diff`, стандартный POSIX-набор;
- (опционально) `python3` для альтернативного ранера с лучшим
  diff-выводом — `run_tests.py`.

## Замечания

1. **Тест `07_effects/04_pure_float_div.lang`** и валидность
   `99_integration/particles.lang` зависят от того, применена ли правка
   A.1 из `consistency_check.md` (float-деление не несёт `@panics`).
   Без этой правки они **не скомпилируются** и тесты упадут.
2. **Точность `float64`**. Тесты с float-выводом проверяют выводы вроде
   `1.5`, `0.5`, `6.28` — значения, которые точно представимы или
   получаются точной арифметикой. Тесты, требующие `printf("%g", ...)` с
   тонкой настройкой формата, не пишутся — это уход от темы.
3. **Перегрузка `print`/`println` по типам** разрешается в codegen
   (`codegen.md` §5.5). Тесты считают, что `println(true)` печатает
   `true`, `println(3.14)` печатает `3.14`, и т.д. Точный формат для
   float описан в codegen — рантайм использует `%g`-подобный формат.
4. **`negative`-тесты** проверяются только по подстроке в stderr, а не
   по точному тексту. Конкретные сообщения компилятора могут эволюционировать.
